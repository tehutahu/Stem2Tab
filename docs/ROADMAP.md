# 開発ロードマップ

この文書は Stem2Tab の実装優先順位を定義する正本です。背景と詳細な技術判断は
[GitHub Issue #1](https://github.com/tehutahu/Stem2Tab/issues/1) を参照してください。

## 現在地

アップロード、非同期ジョブ、Demucs、Basic Pitch、GP5/MusicXML生成、AlphaTab表示までの
Webアプリケーション骨格は実装済みです。一方、生成ファイルの存在だけを確認しており、
実用的なMIDI/TAB品質は保証できていません。

以後はWeb UIの拡張より先に、同じ入力へ複数方式を適用し、結果を再現可能に比較できる
評価基盤を整備します。

## 設計原則

- 分離、F0推定、オンセット検出、ノート分割、リズム量子化、TAB割当を疎結合にする。
- Basic Pitchの最終MIDIを完成品とせず、比較用ベースラインとして扱う。
- 実時間を保つPerformance MIDIと、譜面用に量子化したScore MIDIを分ける。
- 採用方式は聴感だけでなく、ノート精度、リズム誤差、修正操作数で判断する。
- TAB生成はScore MIDIまたは同等の量子化済み内部表現を入力とする。

## 実装フェーズ

### Phase A: 評価基盤

- 共通ノートイベントモデルを定義する。
- MIDI、CSV、JSONの入出力を実装する。
- 同じ音源と設定から比較成果物を生成するベンチマークCLIを実装する。
- onset F1、onset+offset F1、frame F1、オクターブ誤り等を計測する。
- 設定、実行環境、モデル、パラメータを結果へ記録する。

### Phase B: Basic Pitchのベースライン化

- 周波数範囲、閾値、最小音長を設定化する。
- 生の推論出力と変換前後のノートイベントを保存する。
- ベース音域フィルタ、短音除去、同音結合、オクターブ補正を追加する。
- デフォルト設定と調整設定を同じ入力で比較する。

### Phase C: 単旋律パイプライン

- pYIN、PESTO、TorchCREPE等を交換可能なアダプターとして比較する。
- F0、confidence、voiced/unvoiced、onset/offsetを別々に扱う。
- ノート境界と音高系列を系列全体で最適化する。

新しい本番依存関係を追加する前に、候補、導入コスト、ライセンス、実行環境を確認する。

### Phase D: リズム・楽譜化

- 原曲からbeat/downbeatとtempo mapを推定する。
- Performance MIDIとScore MIDIを生成する。
- 3連符、シャッフル、シンコペーション、タイを含む量子化を評価する。

### Phase E: TABと編集UI

- Score MIDIから弦・フレットを割り当てる。
- DPによる運指最適化、MusicXML/GP5出力を実装する。
- 修正UIと修正操作の記録を追加する。

## 評価データの扱い

評価データの数量、区間長、正解データの作成方法、公開データセットの利用範囲は未決定です。
現時点で、プロジェクトが利用できる信頼済みの正解MIDIはありません。
この判断なしに大規模データセットを一括ダウンロードしたり、リポジトリへ同梱したりしません。

最初の評価基盤は、ユーザーが用意する数曲の短い区間を入力できることを優先します。
正解MIDIがない音源でも、複数モデル・複数パラメータの成果物を並べ、聴感と修正量を比較できる
ようにします。絶対評価へ進む前に、次を相談して決定します。

- 曲数と1曲あたりの区間長
- クリーンなBass Stem、原曲、分離Stemのどれを保存するか
- 正解MIDI/TABを誰がどの粒度で作るか
- 公開データセットをダウンロードのみで使うか、派生物を保存するか
- ライセンス、再配布条件、保存容量
- 指弾き、ピック、スラップ、5弦、シャッフル等のカバレッジ

## 最初のマイルストーン

1. 手元音源を指定できるベンチマークCLIを作る。
2. Basic Pitchのデフォルトと複数パラメータ設定を比較する。
3. 追加依存の少ない古典ベースラインを含め、候補方式を同一形式で出力する。
4. 結果を確認した上で、次に導入するモデルと評価データ規模を決める。

このマイルストーンには、大規模公開データセットの取得とWeb UI拡張を含めません。
